# najva-messenger

یک پیام‌رسان خودمیزبان با رمزنگاری سرتاسری — گفت‌وگوی متنی، تماس صوتی و
تصویری، اشتراک فایل رمزشده و اعلان‌های فوری — که کاملاً روی سروری اجرا
می‌شود که خودتان در اختیار دارید.

پیام‌ها پیش از خروج از دستگاه و داخل مرورگر رمز می‌شوند و سرور هیچ‌گاه
کلیدی که بتواند آن‌ها را بخواند در اختیار ندارد؛ فقط بسته‌های کلید
رمزشده و متن رمزشده را نگه می‌دارد. یک دستور روی یک سرور تازه Ubuntu کل
مجموعه را بالا می‌آورد، چهار سؤال می‌پرسد و گواهی TLS می‌گیرد.

## چرا این پروژه ساخته شد

بیشتر پلتفرم‌های چت خودمیزبان شما را وادار به انتخاب می‌کنند: یا نصب در حد
یک دستور است و رمزنگاری فقط در لایه انتقال انجام می‌شود، یا رمزنگاری واقعی
است و نصب یک آخر هفته کلنجار با YAML می‌طلبد. `najva-messenger` برای حالتی
است که هر دو را می‌خواهید — سرویسی که:

- محتوای پیام‌ها و پیوست‌ها را سمت کلاینت رمز می‌کند،
- صدا و تصویر را از SFU و سرور TURN خودتان عبور می‌دهد، و
- با یک `curl | bash` روی Ubuntu 24.04 معمولی نصب می‌شود،

بدون آنکه گفت‌وگوهایتان را به زیرساخت کس دیگری بسپارید.

## سرویس‌ها

پنج کانتینر با Docker Compose کنار هم بالا می‌آیند:

| سرویس | نقش |
| --- | --- |
| `client` | برنامه تک‌صفحه‌ای React + Vite، service worker، صف آفلاین، فارسی/انگلیسی با چیدمان کامل راست‌به‌چپ |
| `server` | API مبتنی بر Express، PostgreSQL از طریق Prisma، وضعیت حضور با Redis، تحویل با Socket.IO |
| `media-server` | mediasoup به‌عنوان SFU برای تماس‌های صوتی و تصویری دونفره و گروهی |
| `turn` | coturn، برای کاربران پشت NAT یا شبکه‌های محدودکننده |
| `nginx` | تنها نقطه ورود؛ TLS را پایان می‌دهد و ترافیک API، WebSocket و فایل‌های ایستا را پراکسی می‌کند |

بخش مدیریت — مدیریت کاربران، سهمیه فضای ذخیره‌سازی، تیکت‌های پشتیبانی و
آمار سرویس — بخشی از همان برنامه وب است و با یک نشانه ادمین محدود می‌شود.

## مدل رمزنگاری

هر پیام با کلیدی مخصوص همان گفت‌وگو رمز می‌شود که تنها در اختیار
شرکت‌کنندگان است. آن کلید با کلید حساب هر عضو بسته‌بندی می‌شود و کلید حساب
از یک کلید رمزگذاری کلید (KEK) مشتق می‌شود که خودش از گذرواژه حساب به دست
می‌آید. سرور فقط بسته‌های رمزشده و یک تأییدکننده ورود را نگه می‌دارد؛ نه
گذرواژه را می‌بیند، نه KEK را، نه متن اصلی را. پیوست‌ها پیش از آپلود رمز
می‌شوند، بنابراین فایل‌های روی دیسک ناخوانا هستند.

Passkey PRF به‌عنوان مسیر دوم بازگشایی برای بازیابی حساب پشتیبانی می‌شود و
اثر انگشت هویتی به دو نفر اجازه می‌دهد یکدیگر را از مسیری بیرون از سرویس
تأیید کنند.

> **توجه:** چون گذرواژه همان چیزی است که کلیدها از آن مشتق می‌شوند، بازنشانی
> گذرواژه گم‌شده از سمت سرور بدون کنار گذاشتن تاریخچه آن حساب ممکن نیست. این
> ویژگیِ طراحی است، نه قابلیتی که جا مانده باشد — به همین دلیل گزینه بازنشانی
> ادمین در منوی `najva` حساب را از نو می‌سازد و گذرواژه را در جای خود تغییر
> نمی‌دهد.

سلسله‌مراتب کامل کلیدها در [docs/ENCRYPTION.md](docs/ENCRYPTION.md) آمده است.

## نصب

### پیش‌نیازها

- Ubuntu 24.04 با یک IP عمومی و دسترسی root
- یک دامنه که به آن IP اشاره کند، در صورتی که HTTPS می‌خواهید (اختیاری)

اگر Docker نصب نباشد، خودکار نصب می‌شود.

### فقط یک دستور

```bash
curl -fsSL https://raw.githubusercontent.com/MatinMHF/najva-messenger/main/install.sh | sudo bash
```

نصب‌کننده پروژه را در `/opt/najva` کلون می‌کند، تمام مقادیر محرمانه فایل
محیطی را تصادفی می‌سازد و چهار سؤال می‌پرسد:

| سؤال | پیش‌فرض | توضیح |
| --- | --- | --- |
| پورت HTTP | `80` | برای پذیرفتن، Enter بزنید |
| پورت HTTPS | `443` | برای پذیرفتن، Enter بزنید |
| نام کاربری و گذرواژه ادمین | `admin` | گذرواژه دوباره پرسیده می‌شود و باید ۸ نویسه یا بیشتر باشد |
| دامنه | *(ندارد)* | خالی بگذارید تا سرویس روی HTTP ساده و روی IP سرور ارائه شود |
| ایمیل Let's Encrypt | `admin@<domain>` | فقط وقتی پرسیده می‌شود که دامنه داده شده باشد |

اگر دامنه بدهید، certbot در حالت standalone گواهی را صادر می‌کند و پیکربندی
Nginx با یک بلوک HTTPS به‌همراه هدایت از HTTP بازنویسی می‌شود. تمدید گواهی با
تایمر خود certbot انجام می‌شود و Nginx را در جا بارگذاری مجدد می‌کند.

اگر صدور گواهی شکست بخورد — DNS هنوز منتشر نشده باشد یا پورت ۸۰ در دسترس
نباشد — نصب روی HTTP ساده کامل می‌شود و همین را به شما اطلاع می‌دهد. گواهی را
هر زمان می‌توان از منوی مدیریت دوباره تلاش کرد.

## مدیریت

نصب‌کننده دستور `najva` را روی سرور قرار می‌دهد. اجرای آن یک منو باز می‌کند:

| گزینه | کاری که می‌کند |
| --- | --- |
| تلاش دوباره برای گواهی SSL | certbot را دوباره اجرا می‌کند؛ اگر بدون دامنه نصب شده باشد، دامنه را هم می‌پذیرد |
| بازنشانی نام کاربری و گذرواژه ادمین | حساب ادمین را از نو می‌سازد (تأیید می‌گیرد) |
| راه‌اندازی مجدد سرویس | `docker compose restart` |
| توقف سرویس | `docker compose down` |
| شروع سرویس | `docker compose up -d` |
| وضعیت | وضعیت کانتینرها، دامنه، وضعیت TLS و پورت‌ها |
| گزارش‌ها | گزارش کانتینرها را دنبال می‌کند |

تلاش دوباره برای گواهی، فایل محیطی، realm سرویس TURN و پیکربندی Nginx را
یک‌جا به‌روزرسانی می‌کند؛ بنابراین انتقال یک سرویسِ فقط‌IP روی یک دامنه به‌جای
ویرایش دستی سه فایل، فقط یک انتخاب از منو است.

## پیکربندی

تمام تنظیمات زمان اجرا در یک فایل `.env` در ریشه مخزن قرار دارد که
نصب‌کننده آن را می‌نویسد و [`.env.example`](.env.example) مستندش می‌کند.
مقادیری که دانستن‌شان ارزش دارد:

| متغیر | توضیح |
| --- | --- |
| `DATABASE_URL`، `REDIS_URL` | رشته اتصال سرویس‌ها |
| `JWT_SECRET`، `JWT_REFRESH_SECRET`، `SERVER_SECRET`، `MEDIA_JWT_SECRET` | چهار کلید امضای مستقل |
| `TURN_SECRET` | باید با `static-auth-secret` در `turn/turnserver.conf` یکسان باشد |
| `VAPID_PUBLIC_KEY`، `VAPID_PRIVATE_KEY` | جفت‌کلید Web Push، که هنگام نصب ساخته می‌شود |
| `CORS_ORIGIN`، `WEBAUTHN_ORIGIN`، `MEDIA_SERVER_PUBLIC_URL` | هر سه، نشانی عمومی همین سرویس هستند |

## توسعه محلی

```bash
git clone https://github.com/MatinMHF/najva-messenger.git
cd najva-messenger
cp .env.example .env
docker compose -f docker-compose.dev.yml up
```

فایل compose توسعه، کدها را mount می‌کند و API و سرور توسعه Vite را با
بارگذاری داغ اجرا می‌کند. آزمون‌ها کنار خود کد در `server/src/__tests__` و
`client/src/**/__tests__` قرار دارند و با `npm test` در هر کدام از دو پوشه
اجرا می‌شوند.

> **توجه:** ایمیج‌های توسعه وابستگی‌ها را هنگام ساخت درون خود می‌پزند، پس بعد
> از افزودن هر بسته npm باید ایمیج را دوباره بسازید؛ وگرنه کانتینر با همان
> وابستگی‌های قبلی کار می‌کند.

## نکات امنیتی

- متن اصلی پیام‌ها و محتوای پیوست‌ها هرگز به سرور نمی‌رسد؛ سرور تنها متن
  رمزشده و کلیدهای بسته‌بندی‌شده را نگه می‌دارد.
- تمام مقادیر محرمانه در `.env` برای هر نصب جداگانه با `openssl rand` ساخته
  می‌شوند؛ هیچ مقدار پیش‌فرضی وجود ندارد که یادتان برود عوضش کنید.
- فایل `.env` با دسترسی `chmod 600` نوشته می‌شود و از کنترل نسخه کنار گذاشته
  شده است.
- اعتبارنامه‌های TURN توکن‌های کوتاه‌عمر HMAC هستند که برای هر تماس ساخته
  می‌شوند، نه یک نام کاربری و گذرواژه ثابت مشترک.
